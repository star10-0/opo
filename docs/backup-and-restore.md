# Backup and Restore Readiness

هذا الدليل يجهّز الفريق لنسخ واستعادة بيانات النظام بدون بناء بنية Backup آلية معقدة.

## ما الذي يجب نسخه احتياطيًا
1. قاعدة البيانات MongoDB كاملة (جميع collections).
2. ملف البيئة الآمن في الخادم (`.env`) لكن خارج Git.
3. أي ملفات مرفوعة مستقبلًا (إن تم اعتمادها لاحقًا).

## تكرار النسخ المقترح
- نسخة يومية (Daily) مع الاحتفاظ بآخر 7 نسخ.
- نسخة أسبوعية (Weekly) مع الاحتفاظ بآخر 4 نسخ.
- نسخة شهرية (Monthly) مع الاحتفاظ بآخر 3–6 نسخ.

## أخذ نسخة احتياطية (mongodump)

### متطلبات
- أدوات MongoDB Database Tools مثبتة على الخادم.
- صلاحية قراءة على قاعدة البيانات.

### أمر نسخ مضغوط
```bash
mongodump --uri="$MONGO_URI" --archive="backup-$(date +%F-%H%M).gz" --gzip
```

### مثال مع مجلد مخصص
```bash
mkdir -p backups
mongodump --uri="$MONGO_URI" --archive="backups/fuel-$(date +%F-%H%M).gz" --gzip
```

## الاستعادة (mongorestore)

> تنبيه: الاستعادة قد تستبدل بيانات حالية. نفذها في نافذة صيانة وبعد تأكيد نسخة حديثة.

```bash
mongorestore --uri="$MONGO_URI" --archive="backups/fuel-2026-01-20-2300.gz" --gzip --drop
```

- `--drop`: يحذف الـ collections الحالية قبل الاستعادة.
- لا تستخدم `--drop` إلا عند الحاجة الواضحة.

## التحقق بعد الاستعادة
1. تشغيل backend والتحقق من endpoint الصحة:
   ```bash
   curl -i http://localhost:5000/api/health
   ```
2. فحص login من الواجهة.
3. التحقق من وجود بيانات محطة ويوم تشغيلي وتقارير.

## ممارسات أمان مهمة
- لا تحفظ النسخ الاحتياطية داخل repository.
- خزّن النسخ في storage آمن مع صلاحيات محدودة.
- طبّق تشفير للنسخ الاحتياطية عند النقل/التخزين الخارجي.
- اختبر restore بشكل دوري على بيئة staging.
